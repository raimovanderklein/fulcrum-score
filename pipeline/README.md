# encounter-bio / pipeline

The prediction pipeline for [encounter.bio](https://encounter.bio). Lives inside the website repo as a subdirectory.

This directory is self-contained: it has its own Makefile, schema, scripts, and data. The website repo's top-level Makefile delegates to it.

## What this is

Trial-level predictions generated by `formula.predict_orr()` from the Generative Geometry framework (van der Klein 2026, [10.5281/zenodo.19408509](https://doi.org/10.5281/zenodo.19408509)) running against a calibration table back-calculated from 183 historical trials with observed ORRs. Every prediction in `docs/v3_scorecard.md` is reproducible by running `make all`.

## Run

From this directory:

```bash
make all       # rebuild everything from source
make clean     # remove generated databases and summaries
make fresh     # clean + all
make sync      # all + run drift check vs website reports.json
```

Or from the website repo root:

```bash
make pipeline-all
make pipeline-fresh
make pipeline-sync
```

## Generated artifacts (gitignored)

- `data/encounter.db` — trial-level prediction database
- `data/gg_systems_workcopy.db` — GG framework work copy with encounter additions
- `data/predictions_summary.json` — numeric summary for the website to read
- `docs/v3_scorecard.md` — generated scorecard (committed as a release snapshot)

## Why predictions_summary.json exists

The website's `data/reports.json` is hand-curated prose: deck copy, headlines, narratives, editorial framing. The pipeline produces `predictions_summary.json` — the *numeric ground truth* for each trial: predicted value, direction, confidence, falsifier, layer ORRs, method, prediction date, plus any matching readout and score.

The website can read `predictions_summary.json` to display "current framework prediction" sections without forcing the prose into a database. The slug field links website report templates to pipeline trial records.

`check_drift.py` compares the two files slug-by-slug and warns if the website prose has drifted from the pipeline numbers (e.g. n_target mismatch, direction mismatch). Run `make sync` before publishing.

## Methodology

See `docs/v3_scorecard.md` for the full methodology section, including:
- The four prediction tiers (calibrated, stage-scaled, single-scaled, novel from components)
- ORR-to-pCR translation for neoadjuvant trials
- Methodology limits (acquired resistance not modeled, depth-2 ceiling clamping, etc.)
- Track record format (backtested vs prospective, never blurred)
